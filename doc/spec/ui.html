User Interface Possibilities

 

Fit implementations may provide any user interface they wish.  Examples include command-line, GUI, and web-based interfaces.  Implementers are encouraged to provide a standard command-line interface, described below.

 

Non-Standard Features and Fixtures

 

If an implementation provides features that are not described in this specification, and if the use of those features results in Fit documents that cannot be read by other compliant Fit implementations, those features are called “non-standard features” and must be disabled by default.  The user must explicitly request non-standard features in order to enable them.  When enabled, the user interface must display a clear message to the user that using non-standard features will prevent the user’s Fit documents from working with other Fit implementations.  Examples of non-standard features include:

 

  • A change to ColumnFixture that allows users to use spaces or arbitrary symbols in column headers
  • An enhancement that allows users to create non-HTML documents
  • A feature that parses graphics embedded in Fit documents

 

Features that do not affect how users create Fit documents need not be disabled and do not require a warning.  Examples include:

 

  • A change that makes creating fixtures easier
  • An interactive editor that creates Fit tables automatically from fixtures
  • An enhanced reporting mode that links all Fit results into a single nicely-formatted report

 

If an implementation bundles fixtures that are not described in this specification, those fixtures must be inaccessible by default.  The user must explicitly enable them using whatever mechanism they use to tell that Fit implementation about their own fixtures.  The fixtures must clearly be documented as being non-standard and users warned that they will have to re-create the fixtures in order to use their Fit documents with other Fit implementations. 

 

The purpose of these requirements is to ensure that users may use their Fit documents to guide porting and rewrite efforts.  An application’s Fit documents provide long-term value that transcends any one implementation.  Disabling non-standard behavior by default helps prevent users from accidentally creating incompatible Fit documents.

 

Recommended Command-Line Interface

 

Fit implementations that provide a command-line interface should (but are not required to) provide the following interface:

 

fit [options] input-file output-file

 

  • Input-file is the HTML file Fit will read.
  • Output-file is the name of the file Fit will create.  If it already exists, it is overwritten.
  • Options are described below.

Spaces, quotes, and other non-alphanumeric characters are handled in implementation-specific ways.

 

Options

 

The command-line should also take at least three options, as follows.  Implementations that support options other than described here are encouraged to follow the GNU standard, described at http://www.gnu.org/prep/standards/standards.html#Command_002dLine-Interfaces.

 

--encoding=charset

 

Charset is the character set (such as UTF-8) used to decode input-file and create output-file.  Legal values for charset are implementation-specific.

Implementation-specific

 

--version

 

Fit implementations should print their name, version, origin, and legal status as described in the GNU standard.  In addition to providing the canonical name of the Fit implementation, it should also provide the version of the Fit specification that it adheres to.  For example:

 

Fit for Befunge v2.42.1

Conforms to Fit Specification v1.2

Copyright (C) 2008 Cunningham & Cunningham, Inc.

License GPLv2+: GNU GPL version 2 or later

This is free software: you are free to change and redistribute it.

There is NO WARRANTY, to the extent permitted by law.

 

(Note that the correct spelling of Fit is “Fit,” not “FIT.”)

 

Implementations should exit without performing any other action when the --version option is encountered.

 

--help

 

Fit implementations should print brief documentation for how to invoke the program as described in the GNU standard.  For example:

 

usage: fit [options] infile outfile

 

Options

 --encoding=CHARSET    read and write files using specified character set

 --version             print version number

 --help                show this help screen

 

See http://fit.c2.com for further information and to report bugs

 

Implementations should exit without performing any other action when the --help option is encountered.

Error Codes

 

Fit should exit with the following error codes:

 

  • When the input file is successfully processed without any test failures or exceptions: zero
  • When the input file is successfully processed, but has failures or exceptions: the sum of failures plus exceptions
  • When the input file could not be processed: -1 (negative one)
  • When the --help or --version option is provided: zero

 

Error Messages

 

Error messages should be prefixed with the name of the program as described in section 4.4 of the GNU standard.  For example:

 

fit: infile.html: file not found


Error messages are otherwise implementation-specific.
